# Chromium Issue #418456081: Оптимизация покадровой перемотки HTMLVideoElement

> **Источник:** [Chromium Issue Tracker #418456081](https://issues.chromium.org/issues/418456081)  
> **Контекст задачи:** См. [README.md](../README.md)  
> **Архитектурный контекст:** См. [archtecture.md](./archtecture.md)

---

## Обзор проблемы

Issue #418456081 касается оптимизации производительности покадровой перемотки (`frame-by-frame seeking`) в `HTMLVideoElement` в Chromium. Проблема связана с тем, что текущая реализация seeking имеет сложность O(n), где n — количество кадров до целевого момента времени, что критично для приложений, требующих покадровой обработки видео.

### Ключевые характеристики проблемы

- **Производительность:** Chromium значительно медленнее Safari при seeking (примерно в 10 раз)
- **Зависимость от структуры видео:** Производительность зависит от количества I-фреймов в видео
- **Критичность:** Для видео с одним I-фреймом seeking в Chromium может занимать ~31.5 мс на кадр, в то время как в Safari ~2.2 мс

---

## Предложенное решение

### Архитектурный подход

Решение должно затрагивать оптимизацию компонентов медиа-пайплайна, описанных в [архитектуре медиа-компонентов](./archtecture.md). Основные направления оптимизации:

#### 1. Оптимизация `media::FFmpegDemuxer`

**Проблема:** Demuxer отвечает за парсинг видео-файла и извлечение кадров. При seeking он должен найти ближайший I-фрейм и декодировать все промежуточные кадры до целевого момента.

**Предлагаемые улучшения:**
- **Кэширование индекса кадров:** Создание и поддержка индекса позиций I-фреймов в видео-файле для быстрого поиска
- **Оптимизация алгоритма поиска:** Использование бинарного поиска вместо линейного сканирования при поиске нужного I-фрейма
- **Предзагрузка метаданных:** Предварительный парсинг структуры видео для построения карты кадров

**Связь с архитектурой:**
> Согласно [архитектуре](./archtecture.md#3-demuxer-демультиплексор), `media::FFmpegDemuxer` используется для стандартного воспроизведения через `src=`. Производительность seeking напрямую зависит от того, как demuxer находит и извлекает нужные кадры.

#### 2. Оптимизация `media::WebMediaPlayerImpl`

**Проблема:** `WebMediaPlayerImpl` обрабатывает запросы на seeking от `HTMLMediaElement` и координирует работу компонентов пайплайна.

**Предлагаемые улучшения:**
- **Асинхронная обработка seeking:** Параллелизация операций поиска и декодирования
- **Буферизация кадров:** Предварительное декодирование соседних кадров для ускорения последующих операций seeking
- **Оптимизация состояния пайплайна:** Минимизация операций reset/flush при последовательных операциях seeking

**Связь с архитектурой:**
> Как описано в [архитектуре](./archtecture.md#2-webmediaplayerimpl), `media::WebMediaPlayerImpl` владеет `media::PipelineController`, который координирует работу `media::DataSource`, `media::Demuxer` и `media::Renderer`. Оптимизация координации этих компонентов критична для производительности seeking.

#### 3. Оптимизация `media::VideoDecoder`

**Проблема:** После нахождения нужного I-фрейма необходимо декодировать все промежуточные кадры до целевого момента.

**Предлагаемые улучшения:**
- **Селективное декодирование:** Пропуск декодирования кадров, которые не требуются для целевого момента (если это возможно)
- **Оптимизация декодирования P- и B-фреймов:** Улучшение алгоритмов декодирования зависимых кадров
- **Использование аппаратных декодеров:** Приоритизация аппаратных декодеров из `media/gpu` для ускорения процесса

**Связь с архитектурой:**
> Согласно [архитектуре](./archtecture.md#6-декодеры), библиотека `media/` содержит аппаратные декодеры в `media/gpu` и программные декодеры в `media/filters`. Декодеры пробуются в порядке, предоставленном через `media::RendererFactory`, и первый успешный используется для воспроизведения.

#### 4. Оптимизация `media::VideoRenderer` и `media::VideoFrameCompositor`

**Проблема:** Рендеринг кадров должен быть синхронизирован с запросами на seeking.

**Предлагаемые улучшения:**
- **Оптимизация callback-механизма:** Улучшение событийно-управляемого интерфейса `media::VideoRendererSink`
- **Кэширование отрендеренных кадров:** Сохранение последних отрендеренных кадров для быстрого доступа при последовательных операциях seeking

**Связь с архитектурой:**
> Как описано в [архитектуре](./archtecture.md#7-videorenderer-и-синхронизация), `media::VideoRenderer` управляет таймингом и рендерингом видео через `media::VideoRendererSink`, который управляется асинхронными callback'ами из `media::VideoFrameCompositor`.

## Поток данных при оптимизированном seeking

```
HTMLMediaElement.seekToNextFrame()
    ↓
media::WebMediaPlayerImpl::Seek()
    ↓
[Кэшированный индекс кадров] → media::FFmpegDemuxer::FindI-Frame()
    ↓
media::VideoDecoder::Decode() [с оптимизацией]
    ↓
media::VideoRenderer::RenderFrame()
    ↓
media::VideoFrameCompositor::OnNewFrame()
```

### Ключевые оптимизации в потоке:

1. **Быстрый поиск I-фрейма** через кэшированный индекс
2. **Параллельное декодирование** нескольких кадров
3. **Минимальные операции reset/flush** в пайплайне
4. **Кэширование результатов** для повторного использования

---

## Ожидаемые улучшения производительности

При успешной реализации предложенных оптимизаций:

- **Сокращение времени seeking:** С ~31.5 мс до ~3-5 мс на кадр (для видео с одним I-фреймом)
- **Улучшение сложности:** С O(n) до O(log n) для поиска I-фрейма
- **Общая производительность покадрового рендеринга:** С O(n²) до O(n log n)

## Технические детали реализации

### Компоненты, требующие изменений

1. **`media/filters/ffmpeg_demuxer.cc`** — добавление индексации кадров
2. **`media/blink/webmediaplayer_impl.cc`** — оптимизация координации seeking
3. **`media/renderers/video_renderer_impl.cc`** — улучшение обработки кадров
4. **`media/base/video_decoder.h`** — расширение API для селективного декодирования

### Интеграция с существующей архитектурой

Все изменения должны быть совместимы с существующим пайплайном воспроизведения:

```
<video> (blink::HTMLMediaElement)
    ↓
blink/public/platform/media/ (media::WebMediaPlayerImpl)
    ↓
media::PipelineController
    ↓
[media::DataSource, media::Demuxer, media::Renderer]
```

Оптимизации не должны нарушать нормальный поток воспроизведения и должны активироваться только при необходимости (например, при частых операциях seeking).
